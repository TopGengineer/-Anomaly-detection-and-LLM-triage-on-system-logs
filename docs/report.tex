\begin{abstract}
\noindent
This report documents an end-to-end anomaly-detection system for system logs,
built on public data only (the \texttt{loghub} HDFS and BGL datasets). The
application domain is cybersecurity operations, but the primary objective is a
technically strong, rigorously validated study of \emph{event/time-series} data.
The system has three parts: (i) an unsupervised \keyword{autoencoder detector}
whose reconstruction error is the anomaly score, benchmarked against an Isolation
Forest baseline; (ii) a \keyword{univariate Hawkes point process} fit by maximum
likelihood to model the self-excitation (clustering) of log events, validated
with the time-rescaling theorem, whose conditional intensity is proposed as a
detector feature; and (iii) an \keyword{LLM triage agent} that deduplicates and
correlates alerts into ranked, explained incidents. Validation is deliberately
strict: a never-shuffled temporal train/test split, precision--recall AUC rather
than accuracy, a headline metric of \emph{false alarms per day at a fixed
detection rate}, and an explicit check for performance drift across the test
period. We report what worked \emph{and what did not}: HDFS detection is a strong
reproduction of known results (PR-AUC $0.987$) rather than a novelty; the Hawkes
branching ratio on BGL is near-critical ($\eta \approx 0.97$--$1.0$), a genuine
quantitative finding, but a single-exponential kernel underfits the deepest
inter-event gaps; the timing feature roughly \emph{halves} false alarms on BGL
though absolute detection there is hard; and extreme-value thresholding, the
principled tool, does not fit HDFS's discrete scores, so a distribution-free
alternative is used there instead. The intellectual honesty of these findings is
treated as a feature of the work, not a defect.
\end{abstract}

\section{Introduction: what, why, and how}
\label{sec:intro}

\paragraph{What.} We build a pipeline that ingests raw system logs and produces a
ranked list of explained security/operations incidents. Concretely: raw logs
$\rightarrow$ structured events (log parsing) $\rightarrow$ features
$\rightarrow$ an unsupervised detector $\rightarrow$ a self-calibrating alert
threshold $\rightarrow$ an LLM triage layer $\rightarrow$ a dashboard. Alongside
the detector runs the mathematical core: a Hawkes process that models the
\emph{timing} of events.

\paragraph{Why.} System logs are the primary telemetry of any distributed
system, and anomalies in them (failures, attacks, cascading faults) are rare,
unlabeled, and clustered in time. Three properties make this a good vehicle for a
rigorous time-series study. First, anomalies are $\sim$0.1--3\% of the data, so
\emph{how} you measure matters enormously---accuracy is meaningless and
precision--recall under class imbalance is mandatory. Second, log events
\emph{cluster}: one failure triggers cascades of related events, which is exactly
the regime a self-exciting point process (Hawkes) is designed for. Third, the
data is non-stationary, so a detector that looks good on an aggregate can be
useless in practice; measuring \emph{drift} is essential. The application is
cybersecurity; the transferable skill is disciplined modeling and validation of
event data.

\paragraph{How.} Everything is unsupervised at training time---labels are used
\emph{only} for evaluation---and everything is validated on a strict temporal
split. We implement the Hawkes machinery ourselves (the standard \texttt{tick}
library has no wheels for modern Python) and, crucially, prove it correct on
simulated data before trusting it on real data. Every non-trivial design decision
in this report was \emph{driven by a measurement}, not assumed; where the data
contradicted a plan, we changed the plan and documented why.

\paragraph{A note on scope and honesty.} A recurring theme is the distinction
between \emph{what has already been done} by the community and \emph{what this
project contributes}. We are explicit about it throughout, and we summarise it in
Section~\ref{sec:related}. In particular, HDFS anomaly detection with count
features and an autoencoder is one of the most reproduced results in the field;
our contribution is not that number but the timing model, the validation
discipline, the drift-aware self-calibrating threshold, and the triage layer.

\section{Background and related work: prior art vs.\ our contribution}
\label{sec:related}

This section separates the state of the art from what we did, component by
component. Table~\ref{tab:priorwork} is the summary; the prose gives the context.

\paragraph{Log parsing.} Turning free-text log lines into a finite vocabulary of
\emph{event templates} is a solved problem. \keyword{Drain} (He et al., 2017) is
a fixed-depth parse-tree parser that is the de-facto standard, distributed in the
\texttt{logparser} toolkit with benchmark configurations per dataset.
\emph{Prior art.} \ours{What we did:} we use Drain unchanged with the published
loghub configuration---parsing is infrastructure here, not a research question,
and using the standard configuration keeps our templates comparable to published
results.

\paragraph{Log anomaly detection.} There is a large literature. Classical
approaches build a per-session \emph{event-count matrix} and apply PCA, Invariant
Mining, or Isolation Forest (the \texttt{loglizer} toolkit, He et al., 2016).
Deep approaches model event \emph{sequences}: \keyword{DeepLog} (Du et al., 2017)
and \keyword{LogAnomaly} (Meng et al., 2019) use LSTMs to predict the next event
and flag surprises; autoencoders on count vectors are a common unsupervised
baseline. On HDFS specifically, these methods routinely achieve F1 / PR-AUC in
the $0.95$--$0.99$ range, and there are dozens of public reproductions.
\emph{Prior art.} \ours{What we did:} we reproduce the count-vector autoencoder
(PR-AUC $0.987$) precisely because it is a known-good baseline---it validates our
pipeline and gives the triage layer real anomalies to consume. We do \emph{not}
claim it as a contribution. Our detection-side contributions are the
\emph{validation protocol} and the \emph{drift analysis} that most reproductions
omit (they typically shuffle and report accuracy/F1).

\paragraph{Point processes for logs.} Hawkes processes (Hawkes, 1971) are
standard in seismology, finance (order-flow, market microstructure), and social
media, and the time-rescaling theorem (Brown et al., 2002) is the standard
goodness-of-fit tool for point processes. Their use as a \emph{feature source for
log anomaly detection} is uncommon---most log-anomaly work discards fine-grained
timing and models only which events occur. \emph{Partial prior art.} \ours{What
we did:} we fit a univariate exponential-kernel Hawkes process to BGL event
times, estimate the branching ratio, validate with time-rescaling, and
\emph{test} whether the resulting self-excitation signal improves detection. This
timing angle---and the honest measurement of whether it helps---is the
intellectual centerpiece.

\paragraph{Adaptive thresholding.} Extreme-Value Theory (EVT) with
Peaks-Over-Threshold and the Generalized Pareto Distribution is classical
(Pickands, 1975; Balkema--de Haan). Applying it to \emph{streaming} anomaly
scores is the \keyword{SPOT}/\keyword{DSPOT} method (Siffer et al., 2017).
\emph{Prior art.} \ours{What we did:} we implement POT/GPD and a streaming
trailing-window variant, and we report a specific, honest negative finding---EVT
assumes a continuous tail, but the HDFS autoencoder scores are strongly discrete,
so GPD extrapolation misbehaves there; we therefore use a distribution-free
rolling quantile on HDFS and reserve EVT for BGL's continuous scores. The
drift-stabilisation \emph{demonstration} on measured drift is ours.

\paragraph{LLM triage.} Using LLMs to summarise or triage alerts is an active,
recent, and largely engineering-driven area. \emph{Emerging prior art.}
\ours{What we did:} we build a triage agent whose novel element is that
\emph{correlation into incidents is grounded in the self-excitation model}---a
cascade of temporally clustered anomalies becomes a single incident---rather than
being an ad-hoc time-window heuristic, and we emit strictly-typed JSON via
structured outputs.

\section{Data: two datasets, and why both are necessary}
\label{sec:data}

We use \emph{two} loghub datasets because they play complementary roles that
neither can fill alone. This was decided by measurement, not preference.

\begin{table}[H]
\centering
\caption{The two datasets and their roles.}
\label{tab:data}
\small
\begin{tabular}{@{}l l l@{}}
\toprule
 & \textbf{HDFS} & \textbf{BGL} \\
\midrule
Role & Detection + triage showcase & Hawkes-process centerpiece \\
Raw size & 11{,}175{,}629 lines & 4{,}713{,}493 lines \\
Templates (Drain) & 48 & 429 \\
Time span & $\sim$1.6 days & $\sim$214.7 days \\
Labels & per \emph{block} (2.93\% anomalous) & per \emph{line} (7.39\% alerts) \\
Timestamp resolution & \textbf{1 second} & \textbf{microsecond} \\
\bottomrule
\end{tabular}
\end{table}

\paragraph{The timestamp-resolution problem (decisive).} A continuous-time Hawkes
process requires a strict ordering of event times. We measured whether the data
provides one:
\begin{itemize}[nosep]
  \item \textbf{HDFS} carries only 1-second timestamps at $\sim$80 events/second.
  As a result, \textbf{98.8\%} of consecutive events in the global stream share a
  timestamp; \emph{within} a single block, 90.5\% still tie, and every block's
  entire life is $\leq 54$\,s (median $7$\,s), occupying $\sim$2 distinct seconds.
  There is essentially no temporal ordering to fit, at any level. Adding
  sub-second jitter would fabricate the very structure the model is meant to
  measure (the branching-ratio estimate would become an artifact of the jitter),
  so we rejected it.
  \item \textbf{BGL} carries genuine \emph{microsecond} timestamps
  (e.g.\ \texttt{2005-06-03-15.42.50.675872}), with $0\%$ ties and events
  naturally spread over seconds---exactly what a continuous-time Hawkes fit needs.
\end{itemize}
Consequently HDFS hosts detection (its clean per-block labels are ideal for
false-alarms-per-day) and BGL hosts the Hawkes work (its timing is the only place
the math is well-posed). Using two independent benchmarks is a strength, not
redundancy.

\section{Log parsing with Drain}
\label{sec:parsing}

Raw lines are unstructured text; the rest of the pipeline needs structure. Drain
mines a fixed vocabulary of templates:
\begin{quote}\small\ttfamily
081109 203515 148 INFO ... PacketResponder 1 for block blk\_388\ldots terminating\\
$\rightarrow$ template ``PacketResponder <*> for block <*> terminating''
\end{quote}
Each line becomes \texttt{(timestamp, event\_id, template, \ldots)}. HDFS lines
additionally carry a \texttt{block\_id} (the session key for its per-block
labels); BGL lines carry an inline alert label and the microsecond timestamp. The
full HDFS log parses to \textbf{48} templates and BGL to \textbf{429}. A coherence
check confirms correctness: the HDFS \texttt{allocateBlock} template occurs
exactly $575{,}061$ times---once per block---matching the label count exactly.
\emph{(Code: \texttt{src/logtriage/parsing/}.)}

\section{Exploratory analysis: the measurements that shaped the design}
\label{sec:eda}

Three findings changed the plan.

\paragraph{(1) Events cluster hard---self-excitation is real.} The Fano factor
(variance/mean of per-minute counts; $=1$ for a Poisson process) is $5.3$ on the
HDFS sample and $698$--$2286$ on BGL. This is the empirical justification for a
Hawkes model \emph{before any fit}. Figure~\ref{fig:eda} (left) shows the HDFS
event timeline: long near-zero baselines punctuated by sharp bursts---the visible
signature of self-excitation.

\paragraph{(2) The HDFS timestamp problem} (Section~\ref{sec:data})---which moved
the Hawkes work to BGL.

\paragraph{(3) Anomaly-rate drift.} On the temporal split (Section~\ref{sec:detect}),
the earlier 70\% of blocks are $3.26\%$ anomalous and the later 30\% are
$2.16\%$---a base rate that is \emph{not} stationary, foreshadowing the drift
results. Figure~\ref{fig:eda} (right) shows the HDFS block-session size
distribution (median 19 events/block), whose long tail is disproportionately the
anomalous blocks.

\section{Detection on HDFS}
\label{sec:detect}

\subsection{Unit, features, and the validation protocol}

Because HDFS labels are per \emph{block} and a block lives $\leq 54$\,s, the
detection unit is the \keyword{block session}: all of a block's events, grouped,
represented as a \textbf{48-dimensional vector of template counts} (the standard
loglizer ``event count matrix''). There are $575{,}061$ blocks.

The validation protocol is the heart of the project:
\begin{itemize}[nosep]
  \item \keyword{Strict temporal split, never shuffled}: blocks ordered by start
  time; earliest 70\% train, latest 30\% test. No test information leaks into
  training, scaling, or thresholds. The split boundary is
  \texttt{2008-11-11 06:00:46}; train $=402{,}543$ blocks ($3.258\%$ anomalous),
  test $=172{,}518$ ($2.158\%$).
  \item \keyword{PR-AUC, not accuracy or ROC}: positives are $\sim$2\%, so
  accuracy $\approx 98\%$ is uninformative and ROC is optimistic under imbalance.
  \item \keyword{Headline metric: false alarms per day at a fixed detection
  rate}, computed on the \emph{actual} elapsed test time (blocks are not uniform
  in time).
  \item \keyword{Drift}: every metric is also reported over consecutive
  chronological slices of the test period.
\end{itemize}

\subsection{Models: baseline vs.\ autoencoder}

Both are unsupervised and scored through the \emph{same} metrics harness.
\begin{itemize}[nosep]
  \item \textbf{Isolation Forest} (baseline): fit on $\log(1+\text{counts})$; the
  anomaly score is $-\,$\texttt{score\_samples}.
  \item \textbf{Autoencoder} (detector): a multilayer perceptron
  $48\!\rightarrow\!32\!\rightarrow\!16\!\rightarrow\!8\!\rightarrow\!16\!\rightarrow\!32\!\rightarrow\!48$
  with ReLU, trained \emph{only on normal train blocks} (labels never enter
  training); per-block reconstruction MSE is the anomaly score. Preprocessing is
  fit on normal train data only, so no test information leaks.
\end{itemize}

\section{The mathematical core: Hawkes process on BGL}
\label{sec:hawkes}

\subsection{Model}

A univariate Hawkes process with exponential kernel has conditional intensity
\begin{equation}
\lambda(t) \;=\; \mu \;+\; \sum_{t_i < t} \alpha\, e^{-\beta (t - t_i)},
\qquad \mu,\alpha,\beta > 0,
\label{eq:intensity}
\end{equation}
so each event raises the instantaneous rate by $\alpha$, decaying at rate
$\beta$. The \keyword{branching ratio}
\begin{equation}
\eta \;=\; \int_0^\infty \alpha\, e^{-\beta s}\, ds \;=\; \frac{\alpha}{\beta}
\end{equation}
is the expected number of \emph{direct} offspring per event; the process is
stationary/subcritical iff $\eta < 1$. This one number summarises how
cascade-prone the stream is.

\subsection{Estimation}

On an observation window $[0,T]$ with events $\{t_i\}$, the log-likelihood is
\begin{equation}
\ell(\mu,\alpha,\beta) \;=\; \sum_i \log \lambda(t_i)\;-\;\int_0^T \lambda(s)\,ds,
\end{equation}
where the integral is closed-form,
$\int_0^T\!\lambda = \mu T + \frac{\alpha}{\beta}\sum_i \bigl(1 - e^{-\beta(T-t_i)}\bigr)$,
and the sum term is computed in $O(n)$ via Ogata's recursion
$A_i = e^{-\beta(t_i - t_{i-1})}(1+A_{i-1})$, $\lambda(t_i)=\mu+\alpha A_i$. We
maximise $\ell$ in log-space (so positivity is automatic) with L-BFGS-B.

\subsection{Correctness before trust}

Because this is the centerpiece, it is \emph{proven on simulated data before
touching BGL} (\texttt{tests/test\_hawkes.py}): we simulate a process with known
$(\mu,\alpha,\beta)$ via Ogata thinning, fit it, and confirm parameter recovery;
and we confirm the goodness-of-fit test \emph{passes} a correct fit and
\emph{rejects} a deliberately wrong model. We implemented all of this directly
with \texttt{scipy} because the \texttt{tick} library has no wheels for modern
Python; the mathematics is library-independent.

\subsection{Validation: the time-rescaling theorem}

If the fitted model is correct, the compensator
$\Lambda(t) = \int_0^t \lambda(s)\,ds$ transforms the events into a unit-rate
Poisson process, so the rescaled inter-event gaps
$\Delta\tau_i = \Lambda(t_i) - \Lambda(t_{i-1})$ are i.i.d.\ $\mathrm{Exp}(1)$.
We test this with a Kolmogorov--Smirnov statistic and, more informatively, a QQ
plot against $\mathrm{Exp}(1)$. A branching ratio is meaningless if the fit fails
rescaling---this is the model-independent check the project's rigor rests on.

\paragraph{Reading.} The \keyword{branching ratio is near-critical everywhere}
($\eta \approx 0.97$--$1.0$): BGL's event stream is strongly, almost
self-sustaining, self-exciting---a precise \emph{estimated} cascade strength, not
just ``bursty.'' The QQ plots (Figure~\ref{fig:hawkes}, right) show the body of
the rescaled-gap distribution on the $\mathrm{Exp}(1)$ diagonal but the
\emph{upper tail} deviating upward: the deepest quiet gaps are longer than a
single exponential predicts. Interpretation: right after a burst the fitted
excitation predicts many events; when silence follows, the integrated intensity
over that gap is large, producing outlying rescaled gaps. The effect is mild on
the calm window (KS $0.095$) and severe on the extreme cascade (KS $0.431$,
$\eta$ pinned at $\sim$1). A single exponential kernel cannot hold both tight
bursts and deep lulls. The mean rescaled gap of exactly $1.0000$ confirms the
compensator is correctly normalised even where the shape deviates; at $10^5$
points the KS test rejects almost any parametric model, so we lean on the QQ
shape and the magnitude, not the $p$-value.

\section{Does the timing signal actually help detection?}
\label{sec:ablation}

This is the question the BGL half exists to answer. On fixed 60-second BGL
windows we detect alert-containing windows three ways with the \emph{same} model
(Isolation Forest) and the \emph{same} temporal split, changing only the
features:
\begin{itemize}[nosep]
  \item \textbf{content}: counts of the top-50 event templates (bag of events, no
  timing);
  \item \textbf{timing}: the per-window Hawkes self-excitation
  $A(t)=\sum_{t_i<t} e^{-\beta(t-t_i)}$ at two decay scales (1\,s, 30\,s) plus the
  event count---computed from event \emph{times only}, hence leakage-free;
  \item \textbf{content+timing}: both.
\end{itemize}

\begin{table}[H]
\centering
\caption{BGL window detection, feature ablation (test split, $4.71\%$ anomalous).}
\label{tab:ablation}
\small
\begin{tabular}{@{}l c c c@{}}
\toprule
Feature set & PR-AUC & Precision @ 90\% recall & False alarms / day \\
\midrule
content & $0.081$ & $0.046$ & $114$ \\
\ours{timing} & \ours{$\mathbf{0.087}$} & \ours{$\mathbf{0.083}$} & \ours{$\mathbf{61}$} \\
content+timing & $0.090$ & $0.071$ & $72$ \\
chance & $0.047$ & --- & --- \\
\bottomrule
\end{tabular}
\end{table}

\paragraph{The honest, nuanced answer.} Absolute detection is \emph{weak} here
($\sim$$0.09$ PR-AUC, $\sim$2$\times$ chance): detecting BGL fault windows at this
resolution is genuinely hard, and we do not oversell it---at 60\,s most alert
windows fold a few alert lines into heavy normal traffic, so the unsupervised
signal is faint. \emph{But} the timing signal is the more useful one and it
\keyword{cuts false alarms}: at 90\% recall, timing roughly \emph{halves} the
false-alarm rate versus content ($114 \rightarrow 61$ per day) and nearly doubles
precision. Figure~\ref{fig:ablation} shows content collapsing to the chance line
at high recall while timing holds $\sim$2$\times$ chance across the operational
range. So the Hawkes self-excitation, as a feature, is the single most useful
signal and moves the metric the project cares about---it just does not, alone,
turn window-level BGL detection into a strong detector. A rigorously-validated
``timing beats content and halves false alarms, on a hard task'' is a real
finding; a manufactured $0.98$ would not be.

\section{Adaptive thresholding: stabilising false alarms under drift}
\label{sec:evt}

\subsection{The problem}

A detector outputs scores; a threshold turns them into alerts. The drift analysis
exposed the real operational weakness: at a \emph{fixed} threshold, the
false-alarm rate wanders as the score distribution drifts, even while
discrimination stays high. On the HDFS autoencoder scores, across a single 5-hour
test window, a fixed threshold's false alarms swing from $3{,}664$ to
$\mathbf{861{,}686}$ per day---flagging up to 100\% of blocks mid-period
(Figure~\ref{fig:evt}).

\subsection{Extreme-Value Theory (the principled tool)}

By the Pickands--Balkema--de Haan theorem, exceedances of a high threshold $t$
converge to a Generalized Pareto Distribution (GPD):
\begin{equation}
\Pr(X - t > x \mid X > t) \;\approx\; \Bigl(1 + \tfrac{\gamma x}{\sigma}\Bigr)^{-1/\gamma}.
\end{equation}
Fitting $(\gamma,\sigma)$ to observed exceedances and inverting for a target tail
probability $q$ gives the self-calibrating alert level
\begin{equation}
z_q \;=\; t + \frac{\sigma}{\gamma}\!\left[\Bigl(\frac{q\,n}{N_t}\Bigr)^{-\gamma} - 1\right],
\end{equation}
with $n$ the sample size and $N_t$ the number of exceedances. Holding $q$ fixed
while re-estimating $(t,\gamma,\sigma)$ from recent data keeps the flagged
rate---hence the false-alarm rate---steady as the distribution moves. We
implement this (\texttt{AdaptivePot}, GPD fit by method of moments for $O(1)$
streaming updates) and verify on simulated data that it controls the exceedance
rate and tracks drift.

\subsection{An honest negative finding, and the fix}

Applied to HDFS, GPD extrapolation misbehaves (thresholds fall below the data
floor and flag everything). The cause is real: the HDFS autoencoder scores are
strongly \emph{discrete}---$58.7\%$ of normal blocks reconstruct to the
\emph{identical} value, because normal HDFS blocks are so repetitive---which
violates GPD's continuous-tail assumption. We therefore use the
distribution-free \keyword{rolling quantile} on HDFS, and reserve EVT for BGL's
\emph{continuous} timing scores, where its extrapolation to small $q$ genuinely
earns its place.

\centering

flag rate $q=3\%$).}

\begin{tabular}{@{}l c c@{}}
\toprule

\midrule

The adaptive threshold holds the alert rate steady at the transparent cost of
recall in periods where anomalies surge past the flag budget---the honest
operating-point tradeoff, now explicit and tunable via $q$.

\section{LLM triage: from alerts to ranked incidents}
\label{sec:triage}

Paging an analyst once per flagged block is unusable. The triage agent reduces
volume, then explains, in three steps:
\begin{enumerate}[nosep]
  \item \keyword{Deduplicate}: flags with the same signature (entity + template
  multiset) collapse into one, with a count.
  \item \keyword{Correlate into incidents}: flags close in time are grouped into a
  single incident. This is the Hawkes idea made operational---self-exciting events
  are causally linked, so a cascade becomes \emph{one} incident, not 200 pages.
  This grounding in the timing model is what distinguishes it from an ad-hoc
  window heuristic.
  \item \keyword{Explain and rank}: each incident is classified into strictly-typed
  JSON (\texttt{priority}, \texttt{category}, \texttt{summary},
  \texttt{recommended\_action}, \texttt{explanation}) and ranked P1$\rightarrow$P4.
  The explainer is pluggable: a deterministic rule-based backend (category from
  template keywords, priority from severity + cascade size) runs self-contained,
  and an \keyword{Anthropic backend} (\texttt{claude-opus-4-8}, JSON-schema
  structured output) drops in when a credential is configured, falling back
  automatically. There is \emph{one} model call per incident, not per raw flag.
\end{enumerate}

On the top-200 flagged HDFS blocks: \textbf{200 raw flags $\rightarrow$ 90
incidents}, prioritised P1(13)/P2(53)/P3(2)/P4(22) across data-loss /
replication / network categories (Figure~\ref{fig:dash}).

\paragraph{Note.} This environment had no model credential, so the demo ran on the
rule-based explainer; the Anthropic path is written to the current API and
activates with a key.

\section{Honest limitations}
\label{sec:limits}

\begin{itemize}[nosep]
  \item \textbf{HDFS detection is a reproduction}, not a contribution; its role is
  validation and substrate, and we say so plainly.
  \item \textbf{BGL window detection is weak in absolute terms} ($\sim$$0.09$
  PR-AUC); the timing result is a \emph{relative} win (fewer false alarms) on a
  hard task.
  \item \textbf{The single-exponential Hawkes kernel underfits} BGL's deepest
  lulls (time-rescaling upper tail); a sum-of-exponentials kernel is the fix.
  \item \textbf{EVT does not fit the discrete HDFS scores}; the rolling quantile is
  used there, and EVT is reserved for continuous scores.
  \item \textbf{The LLM triage backend was not exercised live} (no credential in
  this environment); only the rule-based path ran end-to-end here.
  \item Window size, feature vocabulary, and decay scales are first choices, not
  swept.
\end{itemize}

\section{Conclusion and future work}
\label{sec:conclusion}

We built a complete, unsupervised log-anomaly pipeline with a stochastic-process
core and an LLM triage layer, validated with a discipline (temporal split,
PR-AUC, false-alarms-per-day, drift) that most public reproductions omit. The
mathematical centerpiece---a Hawkes process fit and validated on BGL---yields a
concrete, interpretable finding (near-critical branching ratio) and an honest
account of where a single-exponential kernel breaks. The timing signal it
produces measurably reduces false alarms, even though absolute detection on BGL
is hard. Throughout, we separated \emph{what the community has already
established} from \emph{what we contributed}, and we reported negative and nuanced
results as first-class outcomes.

\paragraph{Future work.} (1) A richer Hawkes kernel (sum-of-exponentials or
power-law) to fix the upper-tail misfit. (2) Use the time-rescaling
\emph{residual} directly as a calibrated timing-anomaly detector, with EVT
thresholding on those continuous scores. (3) Sweep window size and decay scales;
try the autoencoder in place of Isolation Forest for the timing ablation.
(4) Exercise the LLM triage end-to-end with a real model and evaluate incident
quality against held-out labels.
